# Backend PostgreSQL Protocol

## 1. Mission
Ship PostgreSQL data layers that are **typed at the column level, index-justified by EXPLAIN, migration-disciplined, observable under load, and safe to operate at 3 AM**. The skill owns the conventions a team standardizes on — so 12 services don't end up with 12 different migration strategies, 12 different backup scripts, and 12 different opinions about `text` vs `varchar(255)`.

> **Core principle:** If a query is hot, the planner must have a plan for it. Every hot path is `EXPLAIN (ANALYZE, BUFFERS)`-verified before shipping. Every schema change is a numbered migration with a `down` that has been run end-to-end. Every connection from an app is a least-privilege role with no superuser.

## 2. Standards
Every PostgreSQL artifact MUST follow these rules:

- **Right type, first time**: `timestamptz` over `timestamp`, `numeric` over `float` for money, `text` over `varchar(n)` for free-form, `citext` for case-insensitive email, `uuid` + `gen_random_uuid()` for distributed PKs.
- **Identifier convention**: `snake_case` tables and columns (Postgres-idiomatic, recommended). Indexes `<table>_<cols>_idx`. Constraints `<table>_<cols>_key` / `_check` / `_fk`. No quoted `CamelCase` identifiers.
- **Keys**: PK is `NOT NULL`. Use `bigint generated by default as identity` for new surrogate PKs (not legacy `bigserial`). FKs always declare `ON DELETE` and `ON UPDATE` explicitly — no implicit `NO ACTION` deferrals.
- **Migrations are numbered and reversible**: `migrations/NNNN_<slug>.sql` with explicit `-- up` and `-- down` blocks, or Alembic revisions with `upgrade()` + `downgrade()`. Never `Base.metadata.create_all()` in prod. Never edit an applied migration — add a new one.
- **Online DDL by default**: `CREATE INDEX CONCURRENTLY`, `DROP INDEX CONCURRENTLY`, `pg_repack` for rewrites, batched backfill for `ALTER TABLE ... SET NOT NULL` on large tables. Never `ALTER TABLE ... TYPE` to a *different* representation on a hot large table without a dual-write plan.
- **Indexes are EXPLAIN-justified**: Every new index ships with the `EXPLAIN (ANALYZE, BUFFERS)` output (or a runnable equivalent) showing `Seq Scan → Index Scan` or `Sort → Index Only Scan`. No speculative indexes.
- **Least-privilege roles**: Apps connect as a dedicated `app` role with `SELECT/INSERT/UPDATE/DELETE` on the schema's tables, never as `postgres`. Migrations run as a separate `migrator` role with the minimum needed DDL grants.
- **RLS for multi-tenant row scoping**: `ENABLE ROW LEVEL SECURITY` + `FORCE ROW LEVEL SECURITY` on tenant-scoped tables, with policies keyed off `current_setting('app.user_id')` (or equivalent). Don't trust app-layer filters for tenant isolation.
- **psql is the universal client**: every artifact is copy-paste-runnable in `psql`. No GUI-only idioms. Connection via env vars (`PGHOST`, `PGUSER`, `PGDATABASE`, `PGPASSWORD` or `.pgpass` with mode 0600).
- **Backups are restore-tested**: `pg_dump` (logical) or `pg_basebackup` + WAL archiving (PITR). A backup never restored is a backup you don't have. Quarterly restore drills are a standard ticket.
- **Observability is non-optional**: `pg_stat_statements` enabled, `log_min_duration_statement` set (e.g. 500ms), `auto_explain` for prod profiles, structured logs with `application_name` and `statement` tags.
- **Idempotent scripts**: every script that ships is `IF EXISTS` / `IF NOT EXISTS` guarded so reruns are safe. CI runs migrations twice and diffs the resulting schema.

## 3. Workflow Actions

### `/pg scaffold <service>`
Initialize the data layer for a new service.
- Inputs: service name, db name, schema name, engine (Postgres 14 / 15 / 16+), connection env var.
- Outputs: project layout — `migrations/`, `db/init.sql`, `.env.example`, `Makefile` (targets: `db.up`, `db.down`, `db.psql`, `db.seed`, `db.reset`, `db.dump`, `db.restore`, `db.diff`), `docker-compose.yml` (Postgres + Adminer/pgweb), `alembic.ini` (if Alembic chosen), `scripts/check_migrations.sh` (CI helper).
- Pairs with `/fa scaffold` for app-side wiring.

### `/pg connect <env_or_service>`
Diagnose and standardize connection settings.
- Inputs: target env (dev/staging/prod), env var names, secret source (Vault, AWS SM, `.pgpass`).
- Outputs: validated `DATABASE_URL` (libpq URI form), explicit `PGHOST/PGPORT/PGUSER/PGDATABASE/PGPASSWORD` env binding, `sslmode=verify-full` for managed DBs, `.pgpass` template, pool sizing recommendation (e.g. 10-20 per app instance, pgbouncer for >50).
- Includes: pre-flight `psql -c "select 1"` test, SSL cert verification step, latency probe to primary + standbys.

### `/pg settings <service>`
Define and validate database configuration.
- Inputs: Postgres version, workload profile (OLTP / OLAP / mixed), expected connection count, disk budget, replication topology.
- Outputs: `postgresql.conf` baseline (or `ALTER SYSTEM SET ...`) — `shared_buffers`, `work_mem`, `effective_cache_size`, `random_page_cost`, `wal_level`, `max_wal_senders`, statement timeout, lock timeout, idle_in_transaction_session_timeout.
- Rule: pair every setting with its workload rationale and the value Postgres itself computes (`SHOW ... ; pg_settings_reset_val`).

### `/pg model <entity>`
Define a table.
- Inputs: entity name, primary key strategy (surrogate `bigint identity` / `uuid` / natural), columns + types, FK targets, indexes, constraints.
- Outputs: `CREATE TABLE` with typed columns, `NOT NULL` discipline, `CHECK` constraints for value-domain rules, FK with explicit `ON DELETE`/`ON UPDATE`, indexes (PK + secondary + partial as needed), `COMMENT ON` for non-obvious columns.
- Rule: never use `text` for money (use `numeric(p, s)`), never use `timestamp` without tz (use `timestamptz`), never use `ENUM` for sets that will change (use `text` + `CHECK` or a normalized lookup table).
- Output: `migrations/NNNN_create_<entity>.sql` + down migration.

### `/pg schema <scope>`
Schema-level work (rename, partition, restructure, vacuum-driven cleanup).
- Inputs: scope (single table / schema-wide / cross-schema), operation type.
- Outputs: rename plan (Postgres ≥ 9.2 doesn't rewrite on `ALTER TABLE RENAME`), partition plan (range/list/hash), schema split (extract tenant into its own schema), comment sync, vacuum strategy (`pg_repack`, `VACUUM (ANALYZE)`, `pg_stat_user_tables` follow-up).
- Rule: no `ALTER TABLE ... ALTER TYPE` to a *different* representation on a hot large table — design a dual-write path.

### `/pg query <purpose>`
Write a query.
- Inputs: intent (read / aggregate / upsert / delete), tables, predicates, expected return shape, latency budget.
- Outputs: typed `SELECT` / `WITH` / `INSERT ... ON CONFLICT` / `UPDATE ... RETURNING` / `DELETE` using the team's conventions: CTEs for clarity, window functions for top-N-per-group (`row_number()` or `DISTINCT ON`), LATERAL for top-N-per-parent, keyset pagination over OFFSET for large tables.
- Always includes: `EXPLAIN (ANALYZE, BUFFERS)` output (or a runnable shell command to produce it), index recommendation if `Seq Scan` shows on a >10k-row table, parameter binding (`$1, $2`).
- Output: `sql/<purpose>.sql` + `sql/<purpose>.plan.txt` (EXPLAIN output).

### `/pg index <purpose>`
Index a query path.
- Inputs: query, table size, selectivity, sort key, access pattern (range / equality / containment / full-text / array / jsonb).
- Outputs: index DDL with the right access method (`btree` / `gin` / `gist` / `brin` / `spgist`), correct column order for the leading predicate, `INCLUDE` columns only when an index-only scan target exists, partial index when the predicate is selective, expression index when the query always wraps a function (`lower()`, `date_trunc`).
- Always online: `CREATE INDEX CONCURRENTLY` (split out of the migration transaction).
- Always justified: EXPLAIN before/after with row count, ms, buffer hits.

### `/pg explain <query>`
Run and interpret `EXPLAIN (ANALYZE, BUFFERS, VERBOSE)`.
- Inputs: query, optionally `EXPLAIN` settings (analyze / buffers / verbose / format json / serialize).
- Outputs: parsed plan tree with annotated nodes (Seq Scan, Index Scan, Index Only Scan, Hash Join, Nested Loop, Sort, Aggregate), per-node actual vs estimated row counts, buffer hits vs reads, the top 1-2 optimization opportunities with concrete next steps.
- Rule: if `actual rows` differs from `rows=` (planned) by >10x, the planner's stats are stale — recommend `ANALYZE <table>` first.

### `/pg migrate <change>`
Generate a migration (Alembic or raw SQL).
- Inputs: description of the schema change, target environment.
- Outputs:
  - **Alembic**: `alembic/versions/<rev>_<slug>.py` with `upgrade()` + `downgrade()`, autogenerated `--autogenerate` diffed against the actual model state, hand-corrected for `CREATE INDEX CONCURRENTLY` / enum-rewrite / non-trivial type changes.
  - **Raw SQL**: `migrations/NNNN_<slug>.sql` with `-- up` / `-- down` blocks, `IF EXISTS` / `IF NOT EXISTS` in down paths so rollback can re-run.
- Rule: never edit an applied migration. Add a new one. Test the down migration in CI on a fresh DB.

### `/pg migration init`
Initialize the migration tool of choice.
- Inputs: tool (Alembic / sqitch / sqlx / knex / liquibase / flyway / raw).
- Outputs: tool-specific config (`alembic.ini`, `alembic/env.py` configured for async if pairing with `/fa db`), `migrations/` directory, CI script that verifies down-then-up on a fresh DB.
- Rule: don't introduce a second tool. If the repo already has one, use it.

### `/pg seed <dataset_or_env>`
Seed deterministic test data.
- Inputs: target env (dev/test only, never prod), data volume, entity list, constraints (PII, realism).
- Outputs: `INSERT ... ON CONFLICT DO NOTHING` for idempotency, `gen_random_uuid()` + `now()` for portability, batch sizes under 10k rows per statement, FK-aware order (parents before children), `truncate ... restart identity cascade` for full reset.
- Rule: never seed into prod. Production-like fixtures for local dev live in `seeds/dev/` and are gitignored if they contain PII.

### `/pg backup <scope_or_schedule>`
Configure backups.
- Inputs: scope (full / per-db / per-schema), schedule, target (local / S3 / GCS / Azure), retention, encryption.
- Outputs:
  - **Logical**: `pg_dump` (plain or `-Fc` custom) with `--no-owner --no-privileges` for cross-cluster restore, parallel `-j N` for large DBs, optional `gpg -c` encryption before upload.
  - **Physical**: `pg_basebackup` + `archive_command` for WAL archiving → PITR.
  - **Managed**: snapshot-based (RDS, Cloud SQL, Neon) via provider API.
- Always includes: restore dry-run command in the same artifact; restore-time estimate; quarterly drill ticket template.

### `/pg restore <backup_or_point_in_time>`
Restore from a backup or point-in-time target.
- Inputs: backup file or PITR target time, target cluster, target db name, conflict policy (drop / create / refuse).
- Outputs: `pg_restore` command (for `-Fc` dumps) or `pg_basebackup` replay + WAL replay, parallel `-j N` where supported, pre-flight checks (disk space, role existence), `pg_dump --schema-only` of the source to compare against target before promoting.
- Rule: refuse restore to a non-isolated cluster unless the user explicitly confirms. Always capture a pre-restore `pg_dump --schema-only` snapshot.

### `/pg replicate <topology>`
Configure replication.
- Inputs: topology (primary + 1+ sync/async standbys, cascading, logical), write/read split requirement, cross-region, version compatibility.
- Outputs:
  - **Streaming replication**: `primary_slot_name`, `wal_level = replica`, `max_wal_senders`, `hot_standby`, `primary_conninfo` on standbys, `synchronous_standby_names` (if zero data loss required).
  - **Logical replication**: `pgoutput` plugin, publication on primary, subscription on replica, table filtering.
  - **Cascading + cross-region**: `wal_keep_size`, `archive_command` + object storage, replication lag monitoring via `pg_stat_replication` / `pg_replication_slots`.
- Output: topology config + failover plan + lag dashboard queries.

### `/pg monitor <scope>`
Wire database observability.
- Inputs: workload (OLTP / OLAP), SLOs (p95 query time, replication lag, dead tuple ratio), alerting target.
- Outputs:
  - **Stats extension**: `CREATE EXTENSION pg_stat_statements;` per DB; top-N queries by `total_exec_time`, `mean_exec_time`, `calls`.
  - **Logging**: `log_min_duration_statement = 500ms`, `log_lock_waits = on`, `log_temp_files = 0`, JSON logs via `pgaudit` or `log_line_prefix`.
  - **Metrics export**: Prometheus exporter for `pg_stat_*` (key metrics: active connections, dead tuples, replication lag, longest transaction, autovacuum backlog, index hit ratio).
  - **Traces**: `auto_explain` enabled on a sample of sessions; OpenTelemetry DB instrumentation in the app side.
- Output: monitoring config + dashboard JSON (Grafana) + alert rules.

### `/pg vacuum <scope_or_table>`
Vacuum / bloat cleanup.
- Inputs: table(s) or schema, urgency (preventive / corrective), online vs offline.
- Outputs:
  - **Preventive**: `VACUUM (ANALYZE) t;` — never locks, refreshes stats.
  - **Corrective**: `pg_repack` (online, no `ACCESS EXCLUSIVE` lock) when `n_dead_tup / n_live_tup > 0.2` and queries are slow.
  - **Autovacuum tuning**: per-table `ALTER TABLE t SET (autovacuum_vacuum_scale_factor = 0.05)` for hot tables; `pg_stat_user_tables` follow-up.
- Rule: never `VACUUM FULL` in prod (locks the table). Use `pg_repack` or a maintenance window with explicit user ack.

### `/pg secure <scope>`
Lock down authentication and grants.
- Inputs: scope (cluster / db / role), auth method (scram-sha-256 / cert / ldap / oauth via pgbouncer), allowed networks, role catalog.
- Outputs: `pg_hba.conf` rules (no `0.0.0.0/0` ever), `ALTER ROLE ... PASSWORD NULL` for cert-auth roles, explicit `GRANT/REVOKE` per app role (`SELECT/INSERT/UPDATE/DELETE` on needed tables only, never `ALL PRIVILEGES`), `pg_dump --no-privileges` discipline, `ssl = on` + `hostssl ...` rules.
- Rule: app never connects as `postgres`. Always create a per-app `app` role + per-app database, with grants on the application schema only.

### `/pg rls <table>`
Row-Level Security for tenant isolation.
- Inputs: table name, tenant key column, role catalog, policy intent (select / insert / update / delete).
- Outputs: `ALTER TABLE t ENABLE ROW LEVEL SECURITY; ALTER TABLE t FORCE ROW LEVEL SECURITY;` + per-statement `CREATE POLICY` keyed off `current_setting('app.tenant_id', true)::bigint` (or similar) with index on the policy column, `WITH CHECK` for write paths, `USING` for reads, plus a `BYPASSRLS` attribute for the maintenance role only.
- Rule: always `FORCE` (so the table owner also obeys). Always add an index on the policy column. Test with the role that hits the table, not as `postgres`.

### `/pg role <name_or_purpose>`
Create or refactor a role.
- Inputs: name, purpose (app / migrator / reporter / read-replica / maintenance), required grants, password / cert strategy, default GUCs.
- Outputs: `CREATE ROLE` with `LOGIN`, `NOSUPERUSER`, `NOCREATEDB`, `NOCREATEROLE` (unless explicitly needed), explicit `GRANT USAGE ON SCHEMA` + table-level grants + default privileges, `ALTER ROLE ... SET statement_timeout = '5s'` to prevent runaway queries.
- Rule: least privilege. `postgres` is not for app use. Migrations get a separate `migrator` role with DDL grants only on the application schema.

### `/pg extension <name>`
Install / configure a Postgres extension.
- Inputs: extension name (PostGIS, pg_trgm, pgcrypto, uuid-ossp, hstore, pg_stat_statements, pgaudit, postgres_fdw, file_fdw, pg_repack, citext, tablefunc), target schema, version, superuser requirement.
- Outputs: `CREATE EXTENSION IF NOT EXISTS ... ;`, GUCs if needed (`shared_preload_libraries` for `pg_stat_statements`), usage grants to the app role, version pin to avoid surprise major upgrades.
- Rule: list `shared_preload_libraries` changes separately as a `postgresql.conf` step. Don't install extensions the user hasn't explicitly asked for.

### `/pg psql <command_or_session>`
psql one-liner or session.
- Inputs: target query, meta-command (`\dt`, `\d+`, `\df`, `\x`, `\timing`), output format.
- Outputs: copy-paste `psql` command using env vars, redirected to file when output > 50 lines, with `\set ON_ERROR_STOP on` + `\timing on` baked in via `~/.psqlrc`.
- Rule: never embed the password on the command line. Never run `psql` from inside an app's connection pool (pool exhaustion from long psql sessions is the #1 cause of "the app is down").

### `/pg audit <service_or_db>`
Audit an existing PostgreSQL deployment. See §6 for the audit dimensions.

### `/pg tune <query_or_workload>`
Targeted performance tuning.
- Inputs: hot query or workload, target SLO (p95 latency, throughput), current `EXPLAIN (ANALYZE, BUFFERS)` output, table/index stats.
- Outputs: hypothesis (e.g. "this filter is unsargable due to a function wrap"), minimal change (rewritten predicate, new index, `ANALYZE`, setting change), re-run `EXPLAIN` to confirm ms / row-count / buffer-hits improvement, `VACUUM (ANALYZE)` follow-up.
- Rule: change one thing at a time. Capture before/after with the same input dataset.

### `/pg deploy <topology>`
Pre-deployment + post-deployment runbook.
- Inputs: topology (single / primary+replica / multi-region), maintenance window, rollback plan.
- Outputs: pre-deploy checklist (long-running transactions, replication lag, locks, recent vacuum), migration rehearsal on a copy, deploy steps (sequence: migrator role applies DDL → app reconnects with new schema), post-deploy checks (row counts, checksum, SLO burn, pg_stat_statements delta).
- Rule: every DDL change ships with a known-good rollback migration. For breaking schema changes, ship app-side dual-write in a prior release, then enable the migration in a later release.

## 4. Execution Order (Full PostgreSQL Service Cycle)

For a new service's data layer:

1. `/pg scaffold <service>` → project layout + docker-compose
2. `/pg connect <env>` → connection manifest + sslmode
3. `/pg settings <env>` → postgresql.conf baseline
4. `/pg extension <name> [name ...]` → pg_stat_statements, citext, etc. (per service)
5. `/pg migration init` → choose tool, scaffold `migrations/`
6. `/pg model <entity>` × N → tables
7. `/pg migrate <change>` × N → versioned schema changes
8. `/pg role <name>` × N → app role, migrator role, read replica role
9. `/pg secure <scope>` → pg_hba + grants
10. `/pg rls <table>` × N → tenant isolation where needed
11. `/pg index <query>` × N → justified indexes per query
12. `/pg seed <env>` → dev/test data
13. `/pg backup <scope>` → pg_dump / WAL archiving schedule
14. `/pg restore` dry-run → restore-tested verification
15. `/pg replicate <topology>` → if HA is required
16. `/pg monitor <scope>` → stats + metrics + traces
17. `/pg vacuum <scope>` → autovacuum tuning per hot table
18. `/pg query <purpose>` × N → production queries
19. `/pg explain <query>` → per hot query
20. `/pg tune <query>` → per under-SLO query
21. `/pg audit <service>` → pre-launch review
22. `/pg deploy <topology>` → production cutover

> 🛑 **No production rollout without:** migrations rehearsed on a copy, down migrations tested, RLS policies verified with a non-superuser role, `pg_stat_statements` and `pg_repack` available, backup restore drilled, monitoring dashboards live.

## 5. Output Location
All artifacts written under the service's source tree by default. SQL scripts: `migrations/`, `sql/`, `seeds/`, `rls/`, `roles/`, `conf/`, `audit/`, `tune/`, `backup/`. Project-level docs in `/<project>/services/<service>/db/`. Override with `--out=<path>`.

## 6. Audit Workflow
When asked to audit an existing PostgreSQL deployment:

1. **Version & topology**: Postgres version, primary/standby count, replication mode, cluster size. Flag end-of-life versions (PG 12 reached EOL 2024-11-14).
2. **Type discipline**: `timestamptz` over `timestamp`, `numeric` for money, `text` over `varchar(n)` for free-form. Flag every `timestamp without time zone`, every `float` / `double precision` in money columns.
3. **Identifier hygiene**: `snake_case`, no quoted mixedCase. Indexes named `<table>_<cols>_idx`. Constraints named `<table>_<cols>_key/_check/_fk`.
4. **Key discipline**: PK `NOT NULL`, `bigint generated by default as identity` (not `bigserial`), FKs with explicit `ON DELETE`/`ON UPDATE`.
5. **Migration hygiene**: Every schema change has a versioned migration. Flag `create_all()` in prod, hand-edited tables, applied migrations later edited.
6. **Online DDL discipline**: No `ALTER TABLE ... TYPE` to a different representation on hot large tables; `CREATE INDEX CONCURRENTLY` (not in-line); `pg_repack` (not `VACUUM FULL`) for bloat. Flag any `LOCK TABLE ... IN ACCESS EXCLUSIVE` against hot tables in app code paths.
7. **Index justification**: Every index has an `EXPLAIN` showing it is used. `pg_stat_user_indexes.idx_scan = 0` over a long period → flag for drop with `DROP INDEX CONCURRENTLY`.
8. **Role discipline**: Apps connect as non-superuser dedicated roles. `pg_dump --no-privileges` discipline. No `GRANT ALL PRIVILEGES`. No `GRANT ... TO PUBLIC`.
9. **RLS coverage**: Tenant-scoped tables have `ENABLE` + `FORCE` + policies + index on the policy column. Flag tables where the app layer is the only tenant boundary.
10. **Connection security**: `ssl = on`, `pg_hba.conf` scoped to specific subnets, `scram-sha-256` (not `md5`), `hostssl` rules for cross-network connections.
11. **Vacuum / bloat**: `n_dead_tup / n_live_tup` per hot table, `last_autovacuum` recency, `last_analyze` recency. Flag dead_pct > 20% on a hot table without a vacuum plan.
12. **Backups**: `pg_dump` / `pg_basebackup` / WAL archiving present, schedule + retention documented, restore drill within the last 90 days. Flag "backups exist but never restored".
13. **Replication**: `pg_stat_replication` lag monitored, `synchronous_standby_names` documented for zero-data-loss workloads, failover runbook tested.
14. **Observability**: `pg_stat_statements` enabled, `log_min_duration_statement` set, metrics exported, dashboards live, alerts defined for replication lag / connection saturation / dead tuple ratio.
15. **Performance budget**: p95 latency measured and within budget per hot endpoint. Flag hot endpoints with no EXPLAIN-on-record and no recent `pg_stat_statements` evidence.
16. **Connection pooling**: pgbouncer (or equivalent) fronting high-connection apps, mode matches the use case (transaction mode for OLTP, session mode for tooling), pool sizing tuned to `max_connections`.

Output: a report with `Aligned` components and `Violation` instances, each tagged with severity (`P0` blocks launch / `P1` must-fix this quarter / `P2` nice-to-have), a concrete fix, and an effort estimate.

## 7. Hard Rules
- **Never** connect an app as `postgres` or any other superuser.
- **Never** run `CREATE TABLE` / `ALTER TABLE` / `CREATE INDEX` directly against prod outside a migration tool.
- **Never** use `Base.metadata.create_all()` (or any equivalent) in production code paths.
- **Never** edit an applied migration — add a new one.
- **Never** ship an `ALTER TABLE ... TYPE` to a *different* representation on a hot large table without a dual-write plan.
- **Never** ship a `CREATE INDEX` inside the migration transaction for prod — use `CREATE INDEX CONCURRENTLY` and split the file.
- **Never** ship `VACUUM FULL` against a prod table — use `pg_repack` or a maintenance window with explicit user ack.
- **Never** use `timestamp` without `timestamptz` for new columns.
- **Never** use `float` / `double precision` / `real` for money.
- **Never** use `GRANT ALL PRIVILEGES` or `GRANT ... TO PUBLIC` in app-facing roles.
- **Never** trust app-layer filters as the only tenant boundary — use RLS for tenant-scoped tables.
- **Never** ship a migration without a tested `down` (or `downgrade()` for Alembic) that has been run end-to-end on a fresh DB.
- **Never** ship a new index without `EXPLAIN (ANALYZE, BUFFERS)` evidence that it is used by the target query.
- **Always** wrap connection secrets in env vars (or `.pgpass` mode 0600) — never in connection strings stored in git.
- **Always** enable `pg_stat_statements` and `log_min_duration_statement` before the first prod deploy.
- **Always** test the down migration in CI on a fresh DB.
- **Always** include `IF EXISTS` / `IF NOT EXISTS` in the down path so re-runs are safe.
- **Always** keep `pg_hba.conf` scoped to specific subnets — never `0.0.0.0/0`.
- **Always** `FORCE ROW LEVEL SECURITY` on tenant-scoped tables so the table owner also obeys.
- **Always** `ANALYZE` after a large `INSERT/UPDATE/DELETE` or after `pg_repack`, before re-running `EXPLAIN`.
- **Always** ship a restore drill at least once per quarter if the database is business-critical.

# Reference — Type & Schema Design

## Choosing types

| Need | Use | Not |
|---|---|---|
| Money | `numeric(p, s)` (no `float`) | `money`, `float` / `double` / `real` |
| Time | `timestamptz` (always) | `timestamp` (without tz) |
| Booleans | `boolean` | `char(1) 'Y'/'N'` |
| Free text | `text` (length is just a CHECK) | `varchar(255)` |
| Short codes | `varchar(n)` or domain | `text` if always same length |
| UUIDs | `uuid` + `gen_random_uuid()` (PG13+) | `text` |
| JSON | `jsonb` (indexable) | `json` (textual, slower) |
| Single value from enum | `CREATE TYPE ... AS ENUM` or `citext` | `varchar` w/ CHECK |
| Tags / many values | `text[]` or join table | `jsonb` array of strings |
| Counts up to 32k | `smallint` | `integer` |
| Counts up to 2B | `integer` | `bigint` |
| Counts higher | `bigint` | `numeric` |

Default for keys: `bigint generated by default as identity` (preferred over the legacy `bigserial`). Default for time-bound or random PKs: `uuid default gen_random_uuid()`.

## Keys & constraints

- **PK**: always `NOT NULL`. Prefer surrogate `bigint identity` unless you need distributed / merge-friendly PKs (then `uuid`).
- **FK**: always explicitly `ON DELETE ... ON UPDATE ...`. Default `NO ACTION` defers errors at commit, rarely what app code wants.
  - `CASCADE` — child rows die with parent (e.g. line items of an order)
  - `RESTRICT` — block the parent delete (use for billing/membership)
  - `SET NULL` — orphan with null FK (only if `fk IS NULL` is meaningful)
- **Uniqueness**: prefer `UNIQUE` constraints over ad-hoc indexes, so the planner can use them for FK and constraint exclusion.
- **CHECK** constraints for value-domain rules; **NOT NULL** over a sentinel value like `-1` or `''`.

## Naming

Three sane conventions; pick one project-wide and stick:
- `snake_case` tables + columns (Postgres-idiomatic, recommended)
- `PascalCase` (acceptable if ORM like JPA/EF drives)
- Avoid quoted mixedCase ("`UserName`") — breaks tooling and grep.

Indexes: `<table>_<cols>_idx`. Constraints: `<table>_<cols>_key` / `_check` / `_fk`. Don't embed the index type in the name (`*_btree_idx`) — Postgres doesn't guarantee a particular access method.

## Normalization vs denormalization

- 3NF for OLTP schema, until you measure.
- Denormalize only when: read path is hot, write path is rare, and a join can't be avoided cleanly. Always keep the source of truth in normalized tables; denormalized copies updated via trigger or async job.
- Don't add a materialized view until the read query is actually slow.

## Generated columns (PG12+)

Useful when the value is a deterministic function of other columns and you want to index it:
```sql
ALTER TABLE users ADD COLUMN email_lower citext
  GENERATED ALWAYS AS (lower(email)) STORED;
CREATE INDEX users_email_lower_idx ON users (email_lower);
```
`VIRTUAL` (default) is NOT indexed; use `STORED`.

## Common footguns

- `text`/`varchar` have no performance difference in Postgres. Pick by *intent*, not speed.
- `char(n)` pads with spaces and breaks expectation. Avoid.
- `timestamp` (no tz) is almost always wrong. Use `timestamptz`.
- `money` is locale-dependent. Use `numeric(19, 4)`.
- `serial` / `bigserial` are legacy; use `generated ... as identity` for new tables.
- `ENUM` types can't be removed from a column without an enum-rewrite migration. Consider a `varchar` + CHECK if the set will change.

# Reference — Query Patterns

## Reading order in a query

Knowing the order Postgres executes a `SELECT` helps you read EXPLAIN and write queries the planner can shape:

```
FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT
```

That order is also the order Postgres applies statistics / index lookups. A filter in `WHERE` is usually cheaper than the same condition pushed into `HAVING` or `SELECT`.

## CTEs vs subqueries

`WITH ... AS (...)` is mostly syntactic sugar for subquery. Two important exceptions:

- **Recursive CTE** — only way to express tree/walk queries:
  ```sql
  WITH RECURSIVE orgs_tree AS (
    SELECT id, parent_id, name FROM orgs WHERE id = $1
    UNION ALL
    SELECT o.id, o.parent_id, o.name
    FROM orgs o JOIN orgs_tree t ON o.parent_id = t.id
  )
  SELECT * FROM orgs_tree;
  ```
- **`MATERIALIZED` hint** — `WITH t AS MATERIALIZED (...)` forces the planner to compute the CTE once. By default, in PG12+, a non-recursive CTE referenced once is inlined; if you need it computed once, ask for `MATERIALIZED`.

## Window functions

For "give me a top-N per group" without subqueries:
```sql
SELECT *
FROM (
  SELECT e.*,
         row_number() OVER (PARTITION BY user_id ORDER BY created_at DESC) AS rn
  FROM events e
) x
WHERE rn = 1;
```
`row_number()`, `rank()`, `dense_rank()`, `lag()`, `lead()`, `first_value()`, `percentile_cont()`. Use `OVER (... ORDER BY ...)` without `PARTITION BY` for global ranking.

## UPSERT

```sql
INSERT INTO user_settings (user_id, key, value)
VALUES ($1, $2, $3)
ON CONFLICT (user_id, key) DO UPDATE
  SET value = EXCLUDED.value,
      updated_at = now();
```
`EXCLUDED.column` references the row that *would have been inserted*. Don't use `DO NOTHING` blindly — silent skips make replays lie.

## LATERAL joins

For "each row from A, find the top-N matching rows from B":
```sql
SELECT u.id, recent.*
FROM users u
LEFT JOIN LATERAL (
  SELECT * FROM events
  WHERE events.user_id = u.id
  ORDER BY created_at DESC
  LIMIT 5
) recent ON true;
```
This is *the* pattern for top-N-per-parent before `DISTINCT ON` gets you there.

## DISTINCT ON

Postgres-only one-row-per-group shorthand:
```sql
SELECT DISTINCT ON (user_id) user_id, created_at, payload
FROM events
ORDER BY user_id, created_at DESC;
```
Cheaper than `row_number() WHERE rn = 1` in most cases.

## Pivoting / unnesting

`jsonb_agg`, `array_agg`, `string_agg`. To unnest:
```sql
SELECT user_id, tag
FROM posts, unnest(tags) AS tag;
```
Add `WITH ORDINALITY` (PG9.4+) when you care about the index in the array.

## Parameter style in psql

- `psql -v var=value -c "SELECT :'var'"` substitutes a literal.
- `psql -v var=value -c "SELECT :var"` substitutes as a numeric identifier.
- Don't string-concat values into SQL — use `$1, $2` parameter binding or psql variables; otherwise you reintroduce SQL injection.

## Pagination

- Keyset (seek) pagination, not OFFSET, for large tables:
  ```sql
  SELECT id, name FROM users
  WHERE (created_at, id) < ($1, $2)
  ORDER BY created_at DESC, id DESC
  LIMIT 20;
  ```
- OFFSET 1000000 still scans the 1M-skipped rows; keyset uses the index.

# Reference — Performance & Indexing

## First: measure, don't guess.

Before adding any index:
```sql
EXPLAIN (ANALYZE, BUFFERS, VERBOSE) <your query>;
```
Read the plan top-down: each node shows `(actual rows=..., loops=..., actual time=...)`.
If `actual rows` differs from `rows=` (planned) by 10x, the planner's stats are stale → run `ANALYZE <table>` and re-plan.

## Index types

| Type | When |
|---|---|
| `btree` (default) | equality + range + sort on one or a few cols |
| `hash` | equality only, slightly smaller than btree (rarely worth it) |
| `gin` | `jsonb`, `tsvector`, `intarray`, `hstore` |
| `gist` | ranges, geometry (PostGIS), exclusions, trigrams |
| `brin` | huge tables where rows are physically clustered (time series) |
| `spgist` | space-partitioned (e.g. IP CIDR, 2D points) |

Pick by access pattern, not popularity.

## Composite index order

Put the most-selective column *for the common query pattern* first, but remember the index is unusable from the middle: an index on `(a, b, c)` helps queries filtering on `a`, `a+b`, `a+b+c`, but NOT `b` alone or `b+c`.

For a query `WHERE a = ? ORDER BY b`:
- `CREATE INDEX ... ON t(a, b)` — sorted on `b` per `a`, used by the planner.
- `CREATE INDEX ... ON t(a) INCLUDE (b)` — extra columns non-key, useful for index-only scans when `b` is wide.

## Expression / partial indexes

```sql
CREATE INDEX users_active_email_lower_idx
  ON users (lower(email))
  WHERE deleted_at IS NULL;
```
Matches:
```sql
SELECT * FROM users
WHERE deleted_at IS NULL AND lower(email) = lower($1);
```
Indexes are cheap if the predicate is selective (most rows excluded). Don't index low-cardinality columns unless the partial filter makes the predicate narrow.

## ONLINE index creation

`CREATE INDEX CONCURRENTLY` — doesn't take an `ACCESS EXCLUSIVE` lock, so production-safe. Takes 2-3x longer. Can fail and leave an invalid index; re-run it; drop invalid indexes explicitly:
```sql
SELECT indexrelid::regclass FROM pg_index WHERE NOT indisvalid;
DROP INDEX CONCURRENTLY IF EXISTS <name>;
```

## Bloat, vacuum, analyze

- `AUTOVACUUM` is on by default and tuned reasonably.
- `VACUUM (ANALYZE) t;` reclaims tuples and refreshes stats. Use `VACUUM (ANALYZE, VERBOSE) t;` to see what happens.
- `pg_repack` — online rewrite of a bloated table. Use when `n_dead_tup / n_live_tup > 0.2` AND queries slowed down.
- `pg_stat_user_indexes` — `idx_scan = 0` over a long period = unused index. Drop with `DROP INDEX CONCURRENTLY`.

## `pg_stat_statements`

Enable per DB:
```sql
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
```
Then look at:
```sql
SELECT calls, mean_exec_time, rows, query
FROM pg_stat_statements
ORDER BY mean_exec_time DESC LIMIT 20;
```
For "top by total time":
```sql
... ORDER BY total_exec_time DESC LIMIT 20;
```

## Connection / pooling

- Postgres spawns a process per connection; default `max_connections=100`.
- 10-50 connections per app instance is fine; beyond that put **pgbouncer** in front (transaction-pooling mode for OLTP, session-mode if you have prepared statements / temp tables / `LISTEN`).
- Don't `psql` from inside an app's connection pool — pool exhaustion from long-lived psql sessions is the #1 cause of "the app is down".

## Common slow-query causes (start here)

1. **Stale stats** → `ANALYZE`.
2. **Missing index for the predicate** → read the EXPLAIN, see `Seq Scan` on a >10k row table.
3. **Function in WHERE on a column** → `WHERE date(t.created_at) = ...` — wrap as a range: `t.created_at >= ... AND t.created_at < ...` (so index can be used).
4. **Off-by-one index design** — wrong column order for sort, or column not in the index at all.
5. **High row estimates from `LIKE '%foo%'`** → trigram (`pg_trgm`) GIN index if you must.
6. **`SELECT *`** past a UNION where some columns are `jsonb` — fine on size, but watch it on heap.

# Reference — Migration Discipline

## Pick one tool, don't mix

| Tool | Best for | Trade-offs |
|---|---|---|
| **sqitch** | Pure-SQL teams, repeatable per-DB deploys | Verifies against DSN; great for staging mirrors |
| **sqlx-cli** | Rust apps, type-safe SQL files | Compile-time query checks; tied to Rust tooling |
| **flyway / liquibase** | Java-ish teams, multi-engine (PG + MySQL + …) | Heavyweight; XML/YAML can drift from real DB |
| **alembic** | Python (SQLAlchemy) | Python-first, but raw SQL OK |
| **knex / knex-migrate** | Node apps | Lightweight; CLI doubles as a query builder |
| **prisma migrate** | Prisma-only apps | Tied to schema.prisma; harder to drop raw SQL in |
| **raw SQL files** | Tiny services | Name them `NNNN_*.sql`, ship a custom runner |

**Pick what's already in the repo**. If the user has zero migration tooling, suggest sqitch or a numbered `migrations/*.sql` runner — bare SQL without a runner always ends in "who ran what when".

## Safe-ish DDL rules

Postgres is mostly **not** transactional for DDL: most `ALTER TABLE` statements live in their own implicit transaction. Plus a few statements take `ACCESS EXCLUSIVE` locks that block reads/writes.

- `ALTER TABLE ADD COLUMN with DEFAULT` (non-volatile) **since PG11** is metadata-only — no full-table rewrite. (`varchar(255)`, `numeric(10,0)`, `date`, `timestamptz`, etc. are non-volatile defaults.)
- `ALTER TABLE SET DEFAULT`, `NOT NULL`, `TYPE` (in some casts), `DROP COLUMN` — keep ACCESS EXCLUSIVE briefly, no rewrite.
- `ALTER TABLE ALTER TYPE` to a *different* representation (e.g. `varchar → integer` with bad data) — full rewrite.
- `CREATE INDEX CONCURRENTLY` / `DROP INDEX CONCURRENTLY` — no blocking.
- `VACUUM FULL` — locks `ACCESS EXCLUSIVE`; use `pg_repack` instead.

## Migration file skeleton

```sql
-- migrations/0042_add_users_last_login_at.sql
-- up
BEGIN;
ALTER TABLE users
  ADD COLUMN last_login_at timestamptz;

CREATE INDEX CONCURRENTLY users_last_login_at_idx
  ON users (last_login_at)
  WHERE last_login_at IS NOT NULL;
COMMIT;

-- down
BEGIN;
DROP INDEX CONCURRENTLY IF EXISTS users_last_login_at_idx;
ALTER TABLE users DROP COLUMN IF EXISTS last_login_at;
COMMIT;
```

Notes:
- `CREATE INDEX CONCURRENTLY` cannot run inside a transaction. Split:
  ```sql
  -- 0042_add_users_last_login_at.up.sql
  ALTER TABLE users ADD COLUMN last_login_at timestamptz;
  CREATE INDEX CONCURRENTLY users_last_login_at_idx ON users (last_login_at)
    WHERE last_login_at IS NOT NULL;
  -- 0042_add_users_last_login_at.down.sql
  DROP INDEX CONCURRENTLY IF EXISTS users_last_login_at_idx;
  ALTER TABLE users DROP COLUMN IF EXISTS last_login_at;
  ```
- Always include `IF EXISTS` / `IF NOT EXISTS` in `-- down` paths so the rollback can run twice without error.

## Online-table-migration patterns

### Adding a NOT NULL column

1. Add nullable: `ALTER TABLE t ADD COLUMN c int;`
2. Backfill in batches:  `UPDATE t SET c = … WHERE id BETWEEN $1 AND $2;`
3. Add the constraint defensively: `ALTER TABLE t ALTER COLUMN c SET NOT TRUE;` (PG12+ this is fast if the CHECK passes).
4. Add the FK / index as needed.

Validate bloat mid-migration with `pg_stat_user_tables.n_dead_tup`.

### Replacing a column type

Create the new column, dual-write from app, copy data in a batch, swap the application read path, drop the old column. Don't `ALTER TYPE` on a hot large table — full rewrite.

## CI verification

Before merging a migration:
- Run the migration on a fresh DB built from the previous tag.
- Run the down migration on the same DB.
- Diff the resulting schema against the prior one — DDL drift is the #1 migration bug.

If using sqitch, `sqitch verify` does this. If raw SQL, ship a CI script that `pg_dump`s both DBs and `diff -u` the results.

# Reference — Security & RLS

## Roles: least privilege, no superuser

```sql
-- 1. login role for an app
CREATE ROLE app LOGIN PASSWORD :'app_password';
-- 2. role owning the schema
CREATE ROLE app_owner;
GRANT app_owner TO app;  -- membership, not INHERIT-grants
-- 3. grant table-level access to the least schema needed
GRANT USAGE ON SCHEMA app TO app;
GRANT SELECT, INSERT, UPDATE, DELETE
  ON ALL TABLES IN SCHEMA app TO app;
ALTER DEFAULT PRIVILEGES IN SCHEMA app
  GRANT SELECT, INSERT, UPDATE, ON TABLES TO app;
```

Never connect an app as `postgres`. Never grant `ALL PRIVILEGES`. If the app needs `nextval('seq')` for new PKs, grant `USAGE, SELECT ON ALL SEQUENCES ...` too.

## Row-Level Security (RLS)

```sql
ALTER TABLE tickets ENABLE ROW LEVEL SECURITY;
ALTER TABLE tickets FORCE ROW LEVEL SECURITY;  -- even table owner sees RLS

CREATE POLICY tickets_self_select ON tickets
  FOR SELECT
  USING (user_id = current_setting('app.user_id')::bigint);

CREATE POLICY tickets_self_modify ON tickets
  FOR INSERT
  WITH CHECK (user_id = current_setting('app.user_id')::bigint);
```

The app sets the GUC at the start of every connection:
```sql
SELECT set_config('app.user_id', $1, true);
```

`FORCE` makes RLS apply even to the table owner — that's what you want when the role isn't a superuser.

## Common RLS gotchas

- RLS by default applies only to non-superusers. `FORCE ROW LEVEL SECURITY` closes the gap on the table owner.
- Performance: the predicate must be indexable. `current_setting(...)::bigint` is sargable as long as the column is the same type and the cast is stable.
- `BYPASSRLS` (attribute of role) — last resort, only for maintenance role.
- Aggregate functions (`count(*)`, `sum`) over an RLS table still respect RLS, but planner can be surprise-prone. Test with real data.

## `pg_hba.conf` — connection rules

Two basic rules:
- **Local sockets**: `local all postgres peer`
- **Network from app subnet**: `host app_app 10.0.0.0/8 md5` — md5 hashes the password; `scram-sha-256` better (PG14+ default for new installs).

Always:
- Bind to specific subnets, never `0.0.0.0/0`.
- Reject all non-listed IPv6 by default (`host all all ::0/0 reject`).
- Force `ssl = on` and require `hostssl ...` rules when connecting across trust boundaries.

## Connection-string secrets

- Don't put `?password=...` in connection strings stored in git.
- Use env vars: `PGHOST`, `PGUSER`, `PGDATABASE`, and `PGPASSWORD` (or `.pgpass` with mode 0600).
- Use `sslmode=verify-full` against managed DBs (RDS, Cloud SQL, …).

## SQL injection

- Use parameterized queries (`$1, $2`). Period.
- `psql` variables interpolate as literal text; for untrusted input, prefer `PREPARE` + `EXECUTE` and pass via the host app.
- Identifier injection (table/column name can't be parameterized) — quote with `%I` in `format()` or refuse dynamic identifier SQL.

## Audit / logging

For change auditability:
```sql
CREATE TABLE audit_log (
  id bigserial PRIMARY KEY,
  table_name text NOT NULL,
  row_pk jsonb NOT NULL,
  op char(1) NOT NULL,                  -- I/U/D
  before jsonb,
  after jsonb,
  txid bigint NOT NULL DEFAULT txid_current(),
  ts timestamptz NOT NULL DEFAULT now(),
  app_user text DEFAULT current_user
);
```
Wire to a per-table trigger or extension like `pg_audit`. Don't log secrets (passwords, tokens, PII) — hash or truncate.

## Backups and PITR

- `pg_dump` for logical (single-db) backups. Use `--schema-only` for CI reviews.
- `pg_basebackup` for cluster-level physical backups.
- Production: enable WAL archiving + retention for point-in-time recovery.
- Test restore end-to-end. A backup you never restored is a backup you don't have.

# Reference — Operations & Backups

## psql essentials

```bash
# connect with env vars
PGHOST=db.local PGPORT=5432 PGUSER=app PGDATABASE=prod psql

# single command
psql -c "SELECT count(*) FROM events;"

# file in
psql -v ON_ERROR_STOP=1 -f migration.sql

# pipe CSV in/out
psql -c "COPY events TO STDOUT CSV HEADER" > events.csv
psql -c "COPY events FROM STDIN CSV HEADER" < events.csv
```

Useful psql meta-commands:
- `\l` — list databases
- `\dt` — list tables in `search_path`
- `\dt+` — sizes + description
- `\d table` — describe a table
- `\df` — list functions
- `\x` — toggle expanded display
- `\timing` — show query timing
- `\set ON_ERROR_STOP on` — exit on first error (ship this in CI scripts)
- `\echo :SERVER_VERSION_NAME` — introspect server

Triggers `\set` to set in `~/.psqlrc`:
```sql
\set ON_ERROR_STOP on
\timing on
\set show_context_errors on
```

## pg_dump / pg_restore

```bash
# schema only
pg_dump -h $PGHOST -U $PGUSER --schema-only --no-owner $PGDATABASE > schema.sql

# custom-format, parallel, with blobs
pg_dump -h $PGHOST -U $PGUSER -Fc -j 4 -d $PGDATABASE -f prod.dump

# restore
pg_restore -h $PGHOST -U $PGUSER -d $NEWDB -j 4 --no-owner prod.dump
```

Custom-format dumps support `--list` (see contents) and partial restore (`--schema=... --table=...`).

For very large databases, prefer `pg_dump | pigz > bak.gz` over `pg_dump > bak`, and stream directly to object storage to keep the box free of multi-TB files.

## Backups: schedule

- **Hot / huge DB**: WAL archiving via `archive_command` + base backup snapshots. Allows PITR.
- **Small DB**: nightly `pg_dump` + offsite copy. Test restore quarterly.
- **Cross-region**: encrypt the dump (`pg_dump | gpg -c > bak.gpg`) before upload.

## Replication

- **Streaming replication** (`primary → standby(s)`): set `wal_level = replica`, configure `primary_conninfo` on standby.
- **Logical replication** (`pgoutput` plugin): per-table, cross-version, supports filtering. Use for selective subscriptions or upgrades.
- **Synchronous vs async**: synchronous guarantees zero data loss but multiplies primary latency by the network RTT. Use only for hottest rows.

## `pg_stat_activity` & blocking

```sql
SELECT
  pid, usename, application_name,
  state, wait_event_type, wait_event,
  now() - query_start AS duration,
  left(query, 200) AS query
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY duration DESC;
```

Find what blocks what:
```sql
SELECT blocked.pid AS blocked_pid,
       blocking.pid AS blocking_pid,
       blocking.query AS blocking_query
FROM pg_stat_activity blocked
JOIN pg_locks bl ON bl.pid = blocked.pid
JOIN pg_locks kl ON kl.locktype = bl.locktype
  AND kl.database IS NOT DISTINCT FROM bl.database
  AND kl.relation IS NOT DISTINCT FROM bl.relation
  AND kl.page IS NOT DISTINCT FROM bl.page
  AND kl.tuple IS NOT DISTINCT FROM bl.tuple
  AND kl.transactionid IS NOT DISTINCT FROM bl.transactionid
  AND kl.pid != bl.pid
  AND kl.granted
JOIN pg_stat_activity blocking ON blocking.pid = kl.pid
WHERE NOT bl.granted;
```

Terminate a runaway job: `SELECT pg_cancel_backend(<pid>);` (sends SIGINT, transaction rolls back). Use `pg_terminate_backend(<pid>)` for SIGTERM.

## Vacuum / bloat monitoring

```sql
SELECT schemaname || '.' || relname AS table_name,
       n_live_tup, n_dead_tup,
       round(100 * n_dead_tup / NULLIF(n_live_tup, 0), 1) AS dead_pct
FROM pg_stat_user_tables
WHERE n_live_tup > 10000
ORDER BY dead_pct DESC NULLS LAST;
```

If dead_pct > 20% on a hot table: schedule `pg_repack` to compress without lock.

## Connection limits & pooling

```sql
SHOW max_connections;          -- default 100
SELECT count(*) FROM pg_stat_activity;  -- current
```

If app pool oversubscribes (often with pgBouncer misconfig), this is your #1 outage cause. Front-end: **pgbouncer in transaction-pooling mode** for high-connection-count apps; session-mode for tooling (psql), prepared statements, or `LISTEN/NOTIFY`.

## Quick triage checklist

| Symptom | First check |
|---|---|
| App errors "too many connections" | `count pg_stat_activity` + pgbouncer stats |
| One SELECT suddenly slow | `EXPLAIN ANALYZE`, check row estimates vs actual |
| Insert fails with duplicate key | RACE: switch to `ON CONFLICT` |
| Disk pressure | `pg_database_size`, table bloat query |
| Replication lag | `SELECT pg_last_wal_replay_lsn()`, `pg_stat_replication` |
| WAL accumulating | Check `archive_command` is actually writing |
| `relation X does not exist` | `SHOW search_path;` or qualify with `schema.X` |
